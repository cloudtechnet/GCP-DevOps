## IAM Policies & Conditions in GCP 


## 1️⃣ What is an IAM Policy?

An **IAM Policy** in **Google Cloud** defines **who (member) has what access (role) to which resource**.

It is the **main mechanism used to control access in GCP**.

Simple definition:

> **IAM Policy is a set of rules that define which members have which roles on a resource.**

---

## 2️⃣ Structure of an IAM Policy

An IAM policy is made of **bindings**.

Each binding connects:

```
Member + Role + Resource
```

### Components

| Component | Meaning                         |
| --------- | ------------------------------- |
| Member    | User, group, or service account |
| Role      | Collection of permissions       |
| Resource  | Project, VM, bucket, etc        |

---

### Example IAM Policy

Example policy in JSON format:

```json
{
  "bindings": [
    {
      "role": "roles/compute.admin",
      "members": [
        "user:devops@company.com"
      ]
    }
  ]
}
```

Meaning:

* **User:** [devops@company.com](mailto:devops@company.com)
* **Role:** Compute Admin
* **Resource:** Project

Result: The user can **manage VM instances in that project**.

---

# 3️⃣ What are IAM Conditions?

**IAM Conditions** allow administrators to **grant access only when certain conditions are met**.

This provides **fine-grained access control**.

Conditions can restrict access based on:

* Time
* Resource type
* IP address
* Request attributes

Simple definition:

> **IAM Conditions allow access to resources only if specific conditions are satisfied.**

---

# 4️⃣ Real-World Example (Time-Based Access)

### Scenario

A **support engineer** needs temporary access to production servers only during **working hours**.

Instead of giving permanent access, we create a **conditional IAM policy**.

Condition:

```
Allow access only between 9 AM – 6 PM
```

Example condition expression:

```
request.time < timestamp("2026-12-31T18:00:00Z")
```

Result:

✔ Engineer can access servers during allowed time
❌ Access automatically stops after the defined time

This improves **security and control**.

---

# 5️⃣ Real-World Example (DevOps Environment)

### Scenario

A company has **production storage buckets**.

Developers should:

✔ Access storage objects in **development environment**
❌ Not access **production buckets**

IAM condition can restrict access based on **resource name**.

Example condition:

```
resource.name.startsWith("projects/project-id/buckets/dev-")
```

Result:

Developers can access:

```
dev-storage-bucket
```

But cannot access:

```
prod-storage-bucket
```

---

# 6️⃣ Example IAM Policy with Condition

Example policy:

```json
{
  "bindings": [
    {
      "role": "roles/storage.objectViewer",
      "members": [
        "user:developer@company.com"
      ],
      "condition": {
        "title": "OnlyDevBucketAccess",
        "expression": "resource.name.startsWith('projects/_/buckets/dev-')"
      }
    }
  ]
}
```

Meaning:

* Developer can **view storage objects**
* Only in **development buckets**

---

# 7️⃣ Benefits of IAM Conditions

IAM conditions help to:

* Improve **security**
* Implement **temporary access**
* Restrict access based on **resource type**
* Implement **fine-grained access control**

This is commonly used in **enterprise cloud environments**.

---

# 8️⃣ Simple Interview Answer

> IAM Policies define which members have which roles on a specific resource in Google Cloud. IAM Conditions allow administrators to add rules to these policies so that access is granted only when certain conditions are met, such as time restrictions or resource attributes. This provides more secure and fine-grained access control.
