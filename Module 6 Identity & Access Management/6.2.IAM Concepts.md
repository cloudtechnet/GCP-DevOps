## GCP IAM Concepts

In **Google Cloud IAM**, access control is based on **three main concepts**:

1️⃣ **Members**
2️⃣ **Roles**
3️⃣ **Permissions**

These three work together to control **who can access what in Google Cloud**.

Simple formula:

```
Member + Role + Resource = Access Control
```

---

# 1️⃣ Members (Who is accessing)

A **Member** is an **identity that requests access to a resource** in Google Cloud.

Members are also called **Principals**.

### Types of Members in GCP

| Member Type     | Description                   | Example                                                                                             |
| --------------- | ----------------------------- | --------------------------------------------------------------------------------------------------- |
| User            | Individual Google account     | [rajesh@gmail.com](mailto:rajesh@gmail.com)                                                         |
| Google Group    | Group of users                | [dev-team@company.com](mailto:dev-team@company.com)                                                 |
| Service Account | Identity used by applications | [terraform-sa@project.iam.gserviceaccount.com](mailto:terraform-sa@project.iam.gserviceaccount.com) |
| Domain          | Entire organization domain    | company.com                                                                                         |
| All Users       | Anyone on internet            | public access                                                                                       |

---

### Example

Suppose a company has a **DevOps team**.

Members could be:

```
user:devops1@company.com
user:devops2@company.com
group:devops-team@company.com
```

Instead of assigning access to each user individually, we can assign **permissions to the group**.

---

### Service Account Example (Very Important in DevOps)

Service accounts are used by **applications or automation tools**.

Example:

```
Jenkins
Terraform
CI/CD pipelines
Kubernetes workloads
```

Example service account:

```
jenkins-sa@project.iam.gserviceaccount.com
```

This service account can be given permission to:

```
Create VM
Deploy containers
Access storage
```

---

# 2️⃣ Roles (What level of access)

A **Role** is a **collection of permissions**.

Instead of assigning many permissions individually, GCP groups them into **roles**.

Example:

Role → **Compute Admin**

Contains permissions like:

```
compute.instances.create
compute.instances.delete
compute.instances.start
compute.instances.stop
```

---

### Types of Roles in GCP

#### 1️⃣ Basic Roles (Legacy)

| Role   | Access           |
| ------ | ---------------- |
| Owner  | Full control     |
| Editor | Modify resources |
| Viewer | Read-only access |

Example:

```
Viewer → Can see VM details but cannot start or delete VM
```

---

#### 2️⃣ Predefined Roles (Recommended)

Google provides roles specific to services.

Examples:

| Role             | Purpose                 |
| ---------------- | ----------------------- |
| Compute Admin    | Manage virtual machines |
| Storage Admin    | Manage storage buckets  |
| Kubernetes Admin | Manage GKE clusters     |
| Logging Viewer   | View logs               |

Example:

```
roles/compute.admin
roles/storage.objectViewer
```

---

#### 3️⃣ Custom Roles

Organizations can create **custom roles with selected permissions**.

Example:

Custom DevOps role:

```
compute.instances.start
compute.instances.stop
compute.instances.get
```

But **not allowed**:

```
compute.instances.delete
```

This helps implement **least privilege access**.

---

# 3️⃣ Permissions (What actions are allowed)

A **Permission** defines **a specific action that can be performed on a resource**.

Permissions are the **lowest level of access control**.

Example permissions:

| Permission               | Meaning                 |
| ------------------------ | ----------------------- |
| compute.instances.create | Create VM               |
| compute.instances.delete | Delete VM               |
| storage.buckets.create   | Create storage bucket   |
| storage.objects.get      | Read object from bucket |

---

### Important Rule

Users **do not get permissions directly**.

Permissions are granted through **roles**.

Example:

```
User → Role → Permissions
```

---

### Example

Role:

```
Storage Object Viewer
```

Permissions included:

```
storage.objects.get
storage.objects.list
```

Result:

User can:

```
View files in bucket
Download files
```

User cannot:

```
Upload files
Delete files
```

---

# 4️⃣ How Members, Roles, and Permissions Work Together

Example IAM policy:

```
Member: user:rajesh@gmail.com
Role: roles/compute.admin
Resource: Project
```

Meaning:

```
Rajesh can manage all VM instances in that project.
```

Flow:

```
Member (Rajesh)
       ↓
Role (Compute Admin)
       ↓
Permissions (Create, Delete, Start VM)
```

# 6️⃣ Simple Interview Definition

**Members** → Identities that access resources
**Roles** → Group of permissions
**Permissions** → Specific actions allowed on resources

Simple formula:

```
Member + Role = Permissions
```

---

✅ **Example interview answer (very important)**

> In Google Cloud IAM, Members are identities like users, groups, or service accounts. Roles are collections of permissions that define the level of access. Permissions specify the exact actions that can be performed on resources. When a role is assigned to a member, the member automatically receives all the permissions included in that role.
