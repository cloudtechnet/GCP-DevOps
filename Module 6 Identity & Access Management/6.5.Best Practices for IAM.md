## Best Practices for IAM in GCP 

**IAM Best Practices** help organizations secure their **cloud resources by controlling access properly**.
Following these practices reduces **security risks, unauthorized access, and accidental resource deletion**.

---

# 1️⃣ Follow the Principle of Least Privilege

**Least privilege** means giving users **only the permissions they need to perform their job**.

Do not give **full admin access** unless required.

### Example

❌ Bad Practice
Give developer:

```
Compute Admin
```

This allows:

* Create VM
* Delete VM
* Modify VM

✔ Good Practice
Give developer:

```
Compute Instance Viewer
Compute Instance Admin (v1)
```

This limits unnecessary actions.

### Real-World Scenario

A developer only needs to **restart a VM during testing**.

Instead of full admin access, create a **custom role**:

```
compute.instances.start
compute.instances.stop
compute.instances.get
```

This prevents accidental deletion of production VMs.

---

# 2️⃣ Use Groups Instead of Individual Users

Instead of assigning roles to **many users individually**, use **Google Groups**.

### Example

Group:

```
dev-team@company.com
```

Members:

* [dev1@company.com](mailto:dev1@company.com)
* [dev2@company.com](mailto:dev2@company.com)
* [dev3@company.com](mailto:dev3@company.com)

Assign role:

```
Compute Viewer
```

### Benefits

* Easier management
* Centralized access control
* Faster onboarding/offboarding

### Real-World Example

When a new developer joins:

✔ Add them to the **Dev group**
❌ No need to configure IAM again.

---

# 3️⃣ Use Service Accounts for Applications

Applications should **never use personal user accounts**.

Instead use **Service Accounts**.

### Example

Tool:

```
Terraform
```

Service account:

```
terraform-sa@project-id.iam.gserviceaccount.com
```

Role:

```
Compute Admin
```

Terraform now uses this identity to create infrastructure.

### Real-World Example

CI/CD pipeline deploying applications automatically using a service account.

---

# 4️⃣ Avoid Using Basic Roles

Basic roles give **too many permissions**.

| Basic Role | Access            |
| ---------- | ----------------- |
| Owner      | Full access       |
| Editor     | Modify everything |
| Viewer     | Read-only         |

These roles are **not recommended in production environments**.

### Better Approach

Use **Predefined roles**.

Example:

```
Storage Object Viewer
Compute Admin
Kubernetes Admin
```

This provides **fine-grained control**.

---

# 5️⃣ Use IAM Resource Hierarchy Properly

GCP has a **resource hierarchy**:

```
Organization
   ↓
Folder
   ↓
Project
   ↓
Resources
```

Permissions assigned at **higher levels apply to lower levels**.

### Real-World Example

Company structure:

```
Organization
 ├── Finance Folder
 ├── Development Folder
 └── Production Folder
```

Assign:

```
DevOps Admin → Development Folder
```

DevOps team automatically gets access to **all projects inside that folder**.

---

# 6️⃣ Regularly Audit IAM Access

Organizations should **review IAM permissions regularly**.

Use:

* **Cloud Audit Logs**
* **IAM Policy Analyzer**

### Real-World Example

A company finds an employee who left the organization still has **Editor access**.

Removing this access prevents **security risks**.

---

# 7️⃣ Avoid Long-Lived Service Account Keys

Service account keys can be **misused if leaked**.

Better options:

* **Workload Identity**
* **IAM roles**
* **Short-lived credentials**

### Real-World Example

Instead of storing service account keys in GitHub repositories, use **Workload Identity for GKE applications**.

---

# 8️⃣ Use Separate Roles for Different Teams

Each team should get **only the permissions they require**.

Example company setup:

| Team       | IAM Role               |
| ---------- | ---------------------- |
| Developers | Compute Instance Admin |
| DevOps     | Compute Admin          |
| Security   | Logging Viewer         |
| Data Team  | BigQuery Admin         |

This ensures **controlled access across teams**.

---

# Simple Interview Answer

> IAM best practices include following the principle of least privilege, using groups instead of individual users, avoiding basic roles, using service accounts for automation, and regularly auditing permissions. These practices help organizations secure their cloud resources and prevent unauthorized access.
